= Architecture Evaluation: Database Location

This section evaluates the optimal location for storing high-frequency MQTT sensor data based on key technical and architectural metrics.

[cols="15,40,40", options="header"]
|===
| Evaluation Metric
| Local on Smartphone
| Central Server

| *Storage Capacity*
| **Restricted.** Smartphones have limited storage. High-frequency sensor data will quickly consume space, requiring regular compression or deletion of old data.
| **Highly Scalable.** Capable of storing and analyzing vast datasets spanning months or years without worrying about device limits.

| *Data Reliability & Persistence*
| **Vulnerable.** Permanent data loss occurs if the phone is broken or lost.
| **Highly Reliable.** Central backups prevent data loss if a user device breaks.

| *Performance & Latency*
| **Maximum (Low Latency).** Loading data for graphs takes mere milliseconds. Zooming and filtering are instantaneous because there is no network overhead.
| **Network Dependent.** Noticeable latency. Panning and zooming through large datasets might feel sluggish as new data batches must be fetched over the network continually.

| *Compute Load (Efficiency)*
| **Heavy on Device.** The mobile device must process all data locally, which can drain the battery during complex calculations or rendering.
| **Offloaded Compute.** Algorithms like the Douglas-Peucker reduction can run directly on the server. The server sends only simplified data points, making the mobile app extremely lightweight and fast.

| *Offline Capability*
| **Supported.** Once the data is saved on the phone, the user can analyze historical graphs fully offline.
| **Not Supported.** The app cannot display historical graphs without a stable and active internet connection.

| *Security & Privacy*
| **Privacy by Design.** Sensor data never leaves the user's device, ensuring complete data privacy by default.
| **Requires Management.** Data is stored centrally, requiring active security management. However, it ensures data security even if the physical end-device is compromised or lost.

| *Cost & Maintenance*
| **Zero Server Cost.** No ongoing costs for cloud hosting, and no backend infrastructure to maintain.
| **Ongoing Costs.** Requires renting server space, managing security, configuring backups, and developing/maintaining a backend API.

| *Ecosystem & AI Integration*
| **Isolated Silo.** Data is locked on the individual device, making cross-device analysis or external integrations difficult.
| **MCP & AI Ready.** Structured data can be securely exposed to AI agents via an integrated MCP server, enabling intelligent queries and multi-source analysis.
|===

== Decision Taken
The central database is located on a separate server. All data is stored here and made available to the MCP server. In addition, there is a room database on the smartphone, which contains a significantly smaller volume of data. Only current data is stored here, comprising data points that have been simplified using the Douglas-Peulker algorithm. Older data is deleted regularly. If a user wishes to access data that is not stored in the app, they must enable the data connection to the server. The data retrieved from the server is not stored in the room database in order to keep the data volume compact.

If data is transmitted to the smartphone via MQTT, a switch can be activated in the app so that data is transferred to the server database.
